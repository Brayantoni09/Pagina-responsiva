Panorama del ecosistema móvil

Tipos de apps: nativas, híbridas, PWA

Nativas

Las aplicaciones nativas son todas aquellas que están desarrolladas para un sistema operativo en particular como Android o iOS, estas cuentan con una gran ventaja ya que son más visibles y cuentan con más credibilidad dentro del desarrollo móvil por su desarrollo específico. Gracias a esto cuentan cada una con su propia distribución en tienda como Google Play Store y App Store lo que también garantiza el fácil acceso a funcionalidades avanzadas como la cámara, GPS, sensores de movimiento y autenticación biométrica.

Ventajas de apps nativas:

  • Mayor rendimiento
  • Mayor seguridad y estabilidad
  • Acceso total a funcionalidades del dispositivo

Desventajas de apps nativas:

  • Requisitos de aprobación de tiendas de apps
  • Mayor coste de desarrollo
  • Mantenimiento separado para cada plataforma

Híbridas

Las aplicaciones híbridas son una combinación de aplicaciones nativas y aplicaciones web ya que su núcleo se basa en HTML, CSS y JavaScript. Esto permite que con un solo código fuente se pueda adaptar a Android y iOS mediante la utilización de frameworks como React Native, Flutter, Ionic y Apache Cordova. El funcionamiento de una app híbrida se basa en la capacidad de los frameworks de renderizar la interfaz de usuario dentro de un contenedor nativo, lo que les permite acceder a ciertas funcionalidades del dispositivo. Sin embargo, su rendimiento y acceso a hardware específico dependen en gran medida de la optimización del código y de las API nativas disponibles.

Ventajas de apps híbridas:

  • Desarrollo más rápido y económico
  • Un solo código para múltiples plataformas
  • Compatibilidad con tecnologías web

Desventajas de apps híbridas:

  • Menor rendimiento en comparación con las nativas
  • Acceso limitado a hardware
  • Mayor dependencia de frameworks de terceros

PWA

Progressive web app es una autenticidad ya que permite la combinación de sitios web y aplicaciones móviles las cuales son ejecutadas en navegadores como Google Chrome, Safari y Firefox lo cual facilita el acceso a ellas sin la necesidad de descargar los contenidos como lo son en las apps anteriores (Nativas y Híbridas).

Ventajas de PWA:

  • No requiere de instalación
  • Menor coste de desarrollo
  • Compatibilidad con cualquier navegador

Desventajas de PWA:

  • Menor acceso a funcionalidades del hardware
  • Dependencia de los navegadores

Casos de uso comunes en organizaciones

Tomando en cuenta que el uso de estas depende en gran parte de cuál será el uso que se dará y lo que busca la organización al implementar estos tipos de tecnologías de apps. Por ejemplo, si una empresa lo que busca es un monitoreo interno lo más conveniente para esto es app híbrida, por su gran manejo en cualquier dispositivo. Ya que si busca una manera de emplear marketing lo más conveniente es el uso de PWA, ya que el cliente o cualquier persona solo tiene acceso por un navegador a las ofertas que presenta la empresa, lo cual les permite tener una mejor seguridad.

Sistemas operativos móviles

Android e iOS

Android

Es el sistema operativo móvil más popular en la actualidad, ya que ha resaltado mucho en sus funciones y las grandes cantidades de apps a las que tienes acceso. Este es propiedad de Google lo cual permite una mayor compatibilidad con muchas marcas de dispositivos móviles y dependencias de las mismas compañías gracias a su fuente que es de código abierto.

iOS

Este es un sistema operativo diseñado únicamente para dispositivos de Apple, lo cual les permite un gran rendimiento en sus aplicaciones por el acceso a su hardware. De otra manera, este no comparte su núcleo con otras marcas de dispositivos móviles lo que también los convierte en una empresa con alto nivel de seguridad.

Ciclo de vida de una app

Esto inicia con la idea de satisfacer un área de trabajo de cualquier tipo, si no existe la idea de mejorar algún sistema una aplicación nunca será creada. Por lo tanto al tener la idea es importante el llevarla a cabo la cual debe seguir una serie de etapas para un desarrollo exitoso, el cual comienza con:

  • Planificación: Se definen ideas, objetivos y se evalúa la viabilidad técnica. Se documentan requisitos y se asignan recursos.
  • Diseño y prototipado: Se crean maquetas y prototipos interactivos. Se recopila retroalimentación para mejorar el diseño.
  • Desarrollo e implementación: Se programa la app, tanto en frontend como backend. Se integran funcionalidades según la arquitectura definida.
  • Pruebas y control de calidad: Se realizan pruebas unitarias, de integración y beta. Se corrigen errores y se optimiza el rendimiento.
  • Lanzamiento: Se publica la app en tiendas digitales. Se aplican estrategias de marketing para atraer usuarios.
  • Mantenimiento y actualizaciones: Se monitorea el uso, se corrigen fallos y se agregan mejoras. Se adapta la app a nuevas necesidades y tecnologías.

Lenguajes y entornos de desarrollo

Kotlin / Android Studio, Dart / Flutter

Kotlin. Kotlin es un lenguaje de programación de código abierto diseñado para ser simple, elegante y compatible con Java. Se usa principalmente para desarrollar apps Android, aplicaciones web (junto con JavaScript), y sistemas de servidor. Su sintaxis es parecida a Java y C++, pero más concisa y legible. Además, admite programación funcional y reactiva, lo que lo hace versátil y fácil de aprender para quienes ya conocen Java.

Dart. Dart es un lenguaje de programación eficiente y versátil, ideal para crear aplicaciones móviles, web y de escritorio con alto rendimiento y mantenimiento sencillo. Su evolución ha buscado simplificar el desarrollo de interfaces modernas sin perder potencia. Este artículo presenta sus características, historia y ventajas, destacando por qué Dart es una alternativa robusta para quienes buscan mejorar productividad y adaptarse a las demandas actuales del mercado.

Emuladores y depuración básica

Emulador. Un emulador es un programa que simula el funcionamiento de otro dispositivo o sistema operativo dentro de tu computadora.

Depuración básica. Es el proceso de encontrar y corregir errores (bugs) en una aplicación mientras se está desarrollando. Es como mirar “por dentro” del código para entender qué está fallando y por qué.

Arquitecturas y paradigmas

MVC / MVP / MVVM (idea general) y programación event-driven / reactiva

MVC (Modelo-Vista-Controlador). Este patrón divide la aplicación en tres partes:

  • El modelo gestiona los datos y la lógica del negocio.
  • La vista muestra la interfaz al usuario.
  • El controlador actúa como intermediario, recibiendo las acciones del usuario desde la vista, actualizando el modelo y reflejando los cambios en la vista.

MVP (Modelo-Vista-Presentador). Es una evolución del MVC que mejora la separación entre la vista y el modelo:

  • El modelo sigue siendo responsable de los datos.
  • La vista es más pasiva y se limita a mostrar información.
  • El presentador toma el control total de la lógica: recibe eventos de la vista, consulta el modelo y actualiza la vista.

MVVM (Modelo-Vista-ViewModel). Este patrón se enfoca en simplificar el desarrollo de interfaces modernas:

  • El modelo gestiona los datos.
  • La vista muestra la interfaz, usualmente definida en lenguajes de marcado como XML o XAML.
  • El ViewModel conecta ambos mediante enlaces de datos (data binding), transformando la información del modelo en algo que la vista pueda mostrar fácilmente.

Programación event-driven. Es un paradigma donde el flujo del programa está determinado por eventos: clics, entradas de usuario, señales del sistema, etc. Responde a eventos puntuales y ejecuta funciones específicas.

Programación reactiva. Es un paradigma más avanzado que trabaja con flujos de datos asíncronos (streams) y propagación automática de cambios. Gestiona flujos continuos de datos y actualiza automáticamente los componentes que dependen de ellos.

Capas lógicas típicas (UI, dominio, datos) y buenas prácticas de separación

Capas lógicas típicas:

  • Capa de presentación (UI – User Interface): Parte visible para el usuario, encargada de mostrar la información y recibir interacciones (clics, formularios, gestos).
  • Capa de dominio (Lógica de negocio): Contiene las reglas, procesos y decisiones que definen cómo funciona el sistema. Debe ser independiente de la interfaz y de la base de datos.
  • Capa de datos (Data Layer): Gestiona el acceso a fuentes externas como bases de datos, APIs o archivos.

Buenas prácticas de separación:

  • Responsabilidad única: Cada capa debe tener un propósito claro.
  • Comunicación dirigida: Las capas deben comunicarse de forma ordenada.
  • Independencia tecnológica: La lógica de negocio debe funcionar sin depender de frameworks o librerías específicas.
  • Testabilidad: Separar bien las capas facilita escribir pruebas unitarias para cada parte del sistema.
  • Reutilización: Una lógica bien aislada puede usarse en diferentes interfaces (web, móvil, escritorio) sin duplicar código.

Diseño responsivo para la web (porque la evidencia es un sitio)

El diseño responsivo busca que un sitio web se adapte fluidamente a cualquier tamaño de pantalla, desde móviles hasta monitores grandes.

Mobile-first, media queries, Flexbox/Grid, imágenes responsivas

  • Mobile-first: se diseña primero para pantallas pequeñas y luego se escala hacia dispositivos más grandes.
  • Media queries: permiten aplicar estilos CSS según el ancho del dispositivo.
  • Flexbox: sistemas de diseño que organizan elementos de forma flexible y estructurada.
  • Imágenes responsivas: se ajustan automáticamente al tamaño del contenedor usando max-width: 100% y height: auto.

Patrones de navegación en móvil (tab bar, drawer, menús compactos)

En apps móviles, la navegación debe ser intuitiva y ocupar poco espacio.

  • Tab bar: barra inferior con íconos para secciones principales.
  • Drawer: menú lateral que se desliza, ideal para opciones secundarias.
  • Menús compactos: como el menú hamburguesa, que agrupa enlaces en un solo ícono.

Accesibilidad y usabilidad

Diseñar para todos significa incluir a personas con discapacidades visuales, cognitivas o motoras. La accesibilidad mejora la experiencia para todos los usuarios y asegura que el contenido sea entendible y navegable.

Semántica HTML5, alt en imágenes, jerarquía de encabezados, foco y contraste

  • Semántica HTML5: usar etiquetas como <header>, <main>, <section> mejora la estructura y comprensión.
  • Alt en imágenes: describe el contenido visual para lectores de pantalla.
  • Jerarquía de encabezados: usar <h1> a <h6> en orden lógico facilita la navegación.
  • Foco y contraste: asegurar que los elementos sean visibles y navegables con teclado.

Buenas prácticas de texto (legibilidad, tamaño y espaciado en pantallas pequeñas)

  • Tamaño mínimo recomendado: 16px para texto base.
  • Espaciado entre líneas: usar line-height de 1.5 a 1.6 mejora la lectura.
  • Tipografía clara: fuentes como Arial, Roboto o Open Sans son ideales.
  • Uso de clamp(): permite que el tamaño del texto se ajuste dinámicamente.

Seguridad y privacidad (alto nivel)

Las apps móviles deben proteger los datos del usuario desde el diseño, aplicando prácticas seguras y cumpliendo con las normativas de cada país para garantizar la privacidad digital.

Permisos en apps móviles (qué son y ejemplos), HTTPS, manejo responsable de datos

  • Permisos: autorizaciones que la app solicita (ubicación, cámara, contactos).
  • HTTPS: protocolo seguro para transmitir datos cifrados.
  • Manejo responsable de datos: recolectar solo lo necesario, informar al usuario y permitir revocar permisos.

Mención de normativa de datos personales en México (enfoque divulgativo)

México cuenta con leyes robustas para proteger la privacidad digital:

  • LFPDPPP: regula el tratamiento de datos por empresas privadas.
  • LGPDPPSO: aplica a entidades públicas.
  • Derechos ARCO: acceso, rectificación, cancelación y oposición.
  • Aviso de privacidad: documento obligatorio que informa al usuario cómo se usarán sus datos.

Datos y conectividad (alto nivel)

En el desarrollo móvil moderno, los datos y la conectividad son el núcleo de la experiencia del usuario. Ya no basta con que una app funcione; debe hacerlo de forma ágil, segura y resiliente, incluso en condiciones de red limitadas.

APIs REST/JSON y almacenamiento local (nociones: caché, offline-first)

  • APIs REST/JSON: permiten intercambiar datos entre cliente y servidor de forma estructurada.
  • Almacenamiento local: guardar datos en el dispositivo (localStorage, IndexedDB, Cache API).
  • Offline-first: diseño que prioriza la funcionalidad sin conexión, usando Service Workers y caché.

Consideraciones de rendimiento y batería en móvil (idea general)

  • Reducir procesos en segundo plano.
  • Limitar servicios como GPS o cámara cuando no se usan.
  • Usar modos de ahorro de energía.
  • Evitar sobrecalentamiento y carga excesiva.

Ciclo de vida del desarrollo

El desarrollo de una app no termina cuando el código compila. Hay procesos esenciales que garantizan calidad, seguridad y distribución efectiva. Aquí se explican tres pilares fundamentales:

Control de versiones (Git), pruebas básicas en emulador/dispositivo, empaquetado y publicación

Control de versiones (Git). Git es como una “línea del tiempo” para tu código.

  • Permite guardar versiones, colaborar sin conflictos y revertir errores fácilmente.
  • Ideal para trabajar en equipo: cada desarrollador puede crear su propia rama (git branch) y luego unirla (git merge) cuando esté lista.

Pruebas básicas en emulador y dispositivo.

  • Emulador: simula un dispositivo móvil en tu computadora y permite probar rápidamente en diferentes versiones de Android/iOS, resoluciones y condiciones de red.
  • Dispositivo físico: muestra el comportamiento real de la app y es esencial para detectar errores que el emulador no reproduce bien.

Empaquetado y publicación. Una vez superadas las pruebas, se prepara el paquete final para su distribución en tiendas digitales (Google Play, App Store). Este proceso incluye la firma de la app, optimización de recursos y carga en las plataformas correspondientes.
